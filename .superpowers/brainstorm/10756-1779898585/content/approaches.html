3 Approaches: AI Chat Form Builder Rethink

Choose the architecture that fits best — or we can mix elements

A

Patch & Fix (minimal change)

Keep WebSocket subscriptions. Fix the 8 known bugs: ADD_FIELD position, UPDATE_LAYOUT type clash, inspectForm returning live data, race condition, duplicate buildOpLabel, multi-page awareness. Add an undo stack.

Pros

  • Least code change
  • Keeps GraphQL subscription pattern

Cons

  • WS lifecycle bugs will resurface
  • Fundamental split-brain between form state in DB vs live Zustand store
  • Race condition is hard to fully eliminate
B ★ Recommended

HTTP Stream + Tool Redesign

Replace WS subscription with a single POST /api/ai/chat that streams NDJSON back. AI gets a real getField / listFields tool that reads live from the DB. Operations stream in and apply to Zustand with a per-response undo snapshot. Clear responsibility split:

  • REST endpoint — owns the AI call, streaming, DB writes
  • useAIStream hook — owns fetch, parse NDJSON, call applyOp
  • applyOp() — single function owns all store mutations from AI
  • useUndoStack — snapshots before first op, exposes undo()
  • AIEditDrawer — pure UI, no business logic

Pros

  • Eliminates race condition entirely
  • Clear, testable responsibility per unit
  • HTTP is debuggable (curl, network tab)
  • Live form state from DB tool call

Cons

  • New REST endpoint alongside GraphQL
  • Some refactoring effort (~2 days)
C

Agent Loop + Checkpoint Model

Full agentic redesign: AI runs in a server-side loop, reads the live form from DB each step, writes operations as a plan, then executes them. Frontend polls for status. Each AI response is a "checkpoint" — user can jump back to any checkpoint.

Pros

  • Most powerful — AI can do multi-step reasoning
  • Checkpoint history = strong undo
  • No frontend state sync complexity

Cons

  • Large scope — 1-2 weeks
  • Polling feels less live than streaming
  • Overkill for current needs
